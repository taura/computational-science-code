# 追加問題の提案 (タイプ1: もう少し工夫の必要な問題)

各トピックに、既存の最易問題 (`problems/00_*`) の次に置く「少し工夫が要る問題」を1案ずつ提案する。

方針:
- そのトピックまでに学んだ知識だけで解けること。
- 単なる指示行1つの挿入より一歩踏み込み, 範囲計算・複数句・二重ループ・測定の考察などを要するもの。
- 無理に作らない。題材が乏しいトピックは「見送り/限定的」と明記した。
- 番号は `01_` を想定 (既存の最易問題が `00_`)。
- ここでは案と概要のみ。採用するものを選んでもらえれば実ファイル化する。

なお「(2) 多少なりとも意味のある計算をする問題」は別途まとめる予定。下記には結果的に(2)も兼ねるものがある。

## 01_intro — `01_stats` 配列の平均と分散

- **概要**: 配列に入った N 個の数値の平均と分散(または標準偏差)を計算して表示する。
- **やること**: 配列・ループ・関数定義・数値型を総合して使う。要素数 N をコマンドライン引数で受け取り, 動的配列を確保して適当な値で初期化 → 平均を関数で計算 → 分散を計算。
- **工夫ポイント**: 分散の式 (2パス: 平均を出してから二乗和 / 1パスの公式) をどう関数に分けるか。整数と浮動小数点の割り算の違い。
- **言語**: C++ / Fortran 両方。入門の総まとめになる。

## 02_parallel — `01_manual_partition` スレッドで手分けして部分和

- **概要**: `for` 構文をまだ使わず, `omp_get_thread_num()` と `omp_get_num_threads()` だけで配列をスレッドに「手で」分割し, 各スレッドが担当範囲の部分和を計算して表示する。
- **やること**: スレッド数を T, 自分の番号を t として, 担当範囲 `[t*n/T, (t+1)*n/T)` を各自で計算し, その範囲だけを足す。
- **工夫ポイント**: 範囲の割り当て計算 (端数の扱い)。「work sharing 構文 (`for`) が無くても自力で分担できる」ことの体験 = 次トピックの `for` のありがたみにつながる。
- **注意**: 全体の合計を1つの共有変数に足し込むと競合になる (reduction は未習) ので, ここでは各スレッドが自分の部分和を**表示するだけ**に留める。
- **言語**: C++ / Fortran 両方。

## 03_for_collapse — `01_grid2d` 二重ループの並列化 (collapse)

- **概要**: 2次元配列 `a[i][j]` の各要素に `a[i][j] = f(i,j)` (例: `sin`/`cos` や `i*j` など) を代入する二重ループを並列化する。
- **やること**: 外側だけの `parallel for` と `collapse(2)` を比較し, どの (i,j) をどのスレッドが処理したかを観察。外側ループの回数がスレッド数より小さいときに collapse が効くことを確かめる。
- **工夫ポイント**: 2次元配列の確保とアクセス (C++ の `a[i*m+j]` / Fortran の `a(i,j)` と添字順)。collapse 有無の違いの理解。
- **言語**: C++ / Fortran 両方。2次元の格子計算 (ステンシルや画像処理) の入口にもなる。

## 04_schedule — `01_uneven_work` 不均一な仕事の負荷分散

- **概要**: 繰り返しごとに仕事量が大きく違うループ (例: 各 `i` について `i` の試し割りで素数判定, あるいは `0..i` の和) を並列化し, `schedule(static)` と `schedule(dynamic)` の実行時間を比較する。
- **やること**: 既定 (static) では負荷が偏って遅くなること, dynamic で改善することを `time` で確認。チャンクサイズ `dynamic,c` も試す。
- **工夫ポイント**: なぜ static だと遅いのかを自分の言葉で説明する。schedule の選び方の根拠を得る。
- **言語**: C++ / Fortran 両方。素数カウントなら結果も意味がある。

## 05_speedup — `01_scaling_study` 台数効果の飽和を観察し改善する

- **概要**: 各要素が独立な重い計算 (既存の `lin_rec` 系など, reduction 不要なもの) を並列化し, スレッド数を 1,2,4,… と変えて GFLOPS 曲線を描く。
- **やること**: `OMP_PROC_BIND=true` の有無で結果がどう変わるかを比較し, 飽和の理由 (コア数, スレッド配置) を考察。問題サイズ `m`, `n` を変えて, 小さすぎると効果が出ないことも確認。
- **工夫ポイント**: 測定を正しく行う (複数回・平均), グラフから飽和点を読む, 改善策を試す。
- **言語**: C++ / Fortran 両方。
- **備考**: このトピックは本質的に「測定と考察」中心なので, タイプ1としてはこの考察色の濃い1問で十分。これ以上は無理に増やさなくてよい。

## 06_reduction — `01_mean_var` 1つの並列ループで2つの縮約 (平均と分散)

- **概要**: 配列について, 総和 `s` と二乗和 `sq` を**同時に** reduction で求め, 平均と分散を計算する。
- **やること**: `reduction(+:s,sq)` のように複数変数の縮約を書く。あるいは内積 + ノルムでもよい。
- **工夫ポイント**: 複数の reduction を1ループで扱う。浮動小数点の足し込み順序で結果が少しぶれることの確認。
- **言語**: C++ / Fortran 両方。統計量という意味のある計算になる。

## 07_gpu_target — (限定的 / 見送り候補)

- `target` 単体でできることは「文をGPUに飛ばす」程度で, 配列を扱う問題は `map` (09) を待つ必要がある。
- もし作るなら `01_where_am_i`: `omp_is_initial_device()` で「いま GPU で動いているか」を判定して表示し, `OMP_TARGET_OFFLOAD=MANDATORY/DISABLED` で挙動が変わることを確かめる程度。工夫の余地は小さい。
- **提案**: 基本は**見送り**。最易問題 (00) のみで十分とする。

## 08_gpu_teams — `01_teams_quiz` 出力行数を予測する

- **概要**: `target teams distribute (parallel for)` を多重に重ねたプログラムについて, `OMP_NUM_TEAMS=T`, スレッド数 `H`, ループ長 `m`,`n` を与えたとき何行出力されるかを**式で予測**し, `wc -l` で答え合わせする。
- **やること**: どの指示行の下が何回実行されるかを各レベルで考える (cs02 のクイズ形式)。
- **工夫ポイント**: teams / distribute / parallel / for の役割分担の理解。`map` 不要 (印字のみ)。
- **言語**: C++ / Fortran 両方。
- **備考**: 計算としての意味は薄い (印字中心) が, 概念理解の「工夫」問題としては有効。意味のある計算版は 09 以降で。

## 09_gpu_map — `01_vecadd` GPUでベクトル加算 (to/from を正しく使う)

- **概要**: `c[i] = a[i] + b[i]` を GPU で計算し, 結果をホストに戻して検証する。
- **やること**: `map(to: a[0:n], b[0:n])` と `map(from: c[0:n])` を適切に指定。to/from を取り違えると結果が合わないことを体験。
- **工夫ポイント**: どのデータを送る/戻すかの判断, 配列セクション記法。`map(tofrom:)` との違い。
- **言語**: C++ / Fortran 両方。GPUでの意味のある計算 (ベクトル加算) になる。

## 10_gpu_speedup — `01_gpu_scaling` 最適な teams × threads を探す

- **概要**: GPU 版の重い計算について, チーム数とスレッド数 (1 か 32 の倍数) を変えて GFLOPS を測り, 最良の構成を見つける。CPU の最高性能とも比較する。
- **やること**: `OMP_NUM_TEAMS` と `num_threads` を振って表にし, 可視化。
- **工夫ポイント**: 探索のやり方, 32 の倍数制約, CPU/GPU の性能比較の解釈。
- **言語**: C++ / Fortran 両方。

## 11_simd — `01_novec` 自動ベクトル化されないループを omp simd で

- **概要**: そのままでは自動ベクトル化されない (例: ポインタの別名可能性で阻まれる) ループに `#pragma omp simd` (`!$omp simd`) を付けてベクトル化し, `-Minfo`/アセンブリで前後を比較する。
- **やること**: 指示行追加の前後で `-Minfo` の報告や `.s` 中の packed 命令 (`vaddpd` 等) の有無を見比べる。
- **工夫ポイント**: 「なぜコンパイラは勝手にベクトル化しなかったのか」を考える。simd reduction を絡めてもよい。
- **言語**: C++ / Fortran 両方。
- **備考**: タイプ1としてはこの1問で十分。

## 12_simd_intrinsics — `01_map_kernel` ベクトル型で配列の要素ごと計算

- **概要**: 配列に対し `y[i] = f(x[i])` (例: 多項式の Horner 評価や `a*x+b`) を, ベクトル型でまとめて `nl` 個ずつ処理する。
- **やること**: `range`/`loadv`/`storev` と算術演算子を使い, ループを `i += nl` で回す。端数 (n が nl で割り切れない場合) の処理も考える。
- **工夫ポイント**: スカラ↔ベクトルの変換, 端数処理, 全体が SIMD 命令になっているかを `.s` で確認。
- **言語**: **C++ のみ** (ベクトル型拡張は C/C++ 専用)。Fortran は `!$omp simd` で同等のことをする版を付記。

## 13_ilp — `01_tune_nl_mp` SIMD・ILP・マルチコアを総動員して最大性能

- **概要**: ベクトル型の長さ `nl` を 8,16,32,… と変えて命令レベル並列を引き出しつつ, `#pragma omp parallel for` でマルチコアも併用し, 最大 GFLOPS を探す。
- **やること**: `nl` とスレッド数の両方を振って最良点を見つけ, GPU の最高性能と比較する (cs04 の集大成に相当)。
- **工夫ポイント**: なぜ `nl` を増やすと速くなるのか (遅延隠蔽) の理解, 2つの軸の同時最適化。
- **言語**: C++ 中心 (ベクトル型)。Fortran は `!$omp simd` + `!$omp parallel do` 版を付記。
